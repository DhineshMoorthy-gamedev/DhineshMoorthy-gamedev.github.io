Why Addressables?

Unity's Addressables system revolutionizes how we manage and load game assets. Instead of bundling everything into your build, you can:

  • üöÄ Reduce initial app size - Download assets on-demand
  • üîÑ Update content remotely - No app store resubmission needed
  • üíæ Optimize memory - Load and unload assets dynamically
  • üåê Multi-platform support - One system, all platforms
  • üì¶ Efficient bundling - Smart dependency management
üí° Real-World Impact: A mobile game reduced its APK size from 500MB to 50MB and achieved 10x faster downloads by moving textures and audio to Addressables hosted on S3.

Part 1: Setting Up Addressables

Installation

  1. Open Window ‚Üí Package Manager.
  2. Switch the package source to Unity Registry.
  3. Find Addressables and Addressables for Androidand click Install.

Initial Configuration

  1. Go to Window > Asset Management > Addressables > Groups
  2. Click Create Addressables Settings
  3. This creates the AddressableAssetsData folder in your project

Marking Assets as Addressable

You can mark assets as Addressable in :

  • Inspector: Select an asset ‚Üí enable Addressable in the Inspector.
  • Or: Drag & Drop into any group from the Addressables Groups window.

Part 2: Creating Asset Groups

You can start working with Addressables even with a single group. Unity automatically puts all marked assets into the Default Local Group.

When to Create More Groups?

You only need additional groups when you want to:

  • Host assets remotely (AWS, GitHub Pages, Firebase, etc.)
  • Separate frequently vs. rarely loaded content
  • Reduce bundle size by splitting categories

Creating a New Group (Optional)

  1. Open Addressables Groups window
  2. Click Create > Group > Packed Assets
  3. Name it (e.g., ‚ÄúRemoteEnvironment‚Äù or ‚ÄúPlayerAssets‚Äù)
  4. Set Build & Load Paths if you're planning remote hosting
‚ö†Ô∏è Pro Tip: For now, keeping everything in the Default Local Group is perfectly fine‚Äîespecially for small projects and early development.

Part 3: Asset Loading Code

Asset Loading

AddressableOnGUILoader.cs 
using Unity.Cinemachine;
using UnityEngine;
using UnityEngine.AddressableAssets;
using UnityEngine.ResourceManagement.AsyncOperations;

public class AddressableOnGUILoader : MonoBehaviour
{
    public AssetReference environmentAssetReference;
    public AssetReference playerAssetReference;

    private AsyncOperationHandle<GameObject> loadHandle;
    private bool isDownloading = false;

    private bool envDownloaded = false;
    private bool playerDownloaded = false;

    public CinemachineCamera CinemachineCamera;
    public GameObject ui;
    public VirtualInput VirtualInput;
    public MobileDisableAutoSwitchControls MobileDisableAutoSwitchControls;

    void OnGUI()
    {
        GUIStyle labelStyle = new GUIStyle(GUI.skin.label)
        {
            fontSize = 18,
            normal = { textColor = Color.white }
        };

        GUIStyle buttonStyle = new GUIStyle(GUI.skin.button)
        {
            fontSize = 20
        };

        GUILayout.BeginArea(new Rect(20, 20, 400, 300));
        GUILayout.Label("Addressables Loader", labelStyle);
        GUILayout.Space(10);

        // Hide buttons during download
        if (!isDownloading)
        {
            if (!envDownloaded)
            {
                if (GUILayout.Button("Download Environment", buttonStyle, GUILayout.Height(50)))
                {
                    StartDownload(environmentAssetReference, "Environment");
                }
            }

            if (!playerDownloaded)
            {
                if (GUILayout.Button("Download Player", buttonStyle, GUILayout.Height(50)))
                {
                    StartDownload(playerAssetReference, "Player");
                }
            }
        }

        // After downloads
        if (envDownloaded || playerDownloaded)
        {
            string downloaded = "";
            if (envDownloaded && playerDownloaded) downloaded += "Downloaded: Environment, Player";

            GUILayout.Label(downloaded, labelStyle);
        }

        GUILayout.EndArea();
    }

    private void StartDownload(AssetReference reference, string name)
    {
        isDownloading = true;

        loadHandle = reference.LoadAssetAsync<GameObject>();
        loadHandle.Completed += (op) => OnLoaded(op, name);
    }

    private void OnLoaded(AsyncOperationHandle<GameObject> handle, string name)
    {
        isDownloading = false;

        if (handle.Status == AsyncOperationStatus.Succeeded)
        {
            GameObject obj = Instantiate(handle.Result);

            if (name == "Environment")
                envDownloaded = true;

            if (name == "Player")
            {
                playerDownloaded = true;
                CinemachineCamera.Follow = obj.transform.Find("PlayerCameraRoot");

#if UNITY_ANDROID
                ui.gameObject.SetActive(true);
                VirtualInput.StarterAssetsInputs = obj.GetComponent<StarterAssetsInputs>();
                MobileDisableAutoSwitchControls.playerInput = obj.GetComponent<UnityEngine.InputSystem.PlayerInput>();
#endif
            }

        }
        else
        {
            Debug.LogError($"Failed to load: {name}");
        }
    }
}
üí° Best Practice: Always use AssetReference in the Inspector instead of string addresses. This provides compile-time safety and prevents typos.

Part 4: Setting Up Remote Hosting

Option 1: AWS S3 Setup

Step 1: Create S3 Bucket

  1. Go to AWS Console ‚Üí S3
  2. Click Create bucket
  3. Name: your-game-addressables
  4. Region: Choose closest to your players
  5. Uncheck "Block all public access" (we'll set specific permissions)
  6. Create bucket

Step 2: Configure Bucket Policy

S3 Bucket Policy JSON 
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": "*",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::your-game-addressables/*"
        }
    ]
}

Step 3: Enable CORS

S3 CORS Configuration 
[
{
    "AllowedHeaders": [
        "*"
    ],
    "AllowedMethods": [
        "GET",
        "HEAD"
    ],
    "AllowedOrigins": [
        "*"
    ],
    "ExposeHeaders": []
}
]

üí° Tip: Don‚Äôt Forget CORS for WebGL!

If you're hosting Addressables on AWS S3 (or any remote server), your WebGL build will fail to load assets unless CORS is enabled.

  • Allow GET and HEAD methods
  • Set Access-Control-Allow-Origin: *

Without this, browsers will block downloads and show "No 'Access-Control-Allow-Origin' header present".

Only WebGL requires this ‚Äî not Android or PC.

Option 2: GitHub Pages Setup

Step 1: Create Repository

  1. Create new GitHub repository: game-addressables
  2. Make it public (required for GitHub Pages)
  3. Clone to local machine

Step 2: Enable GitHub Pages

  1. Go to repository Settings ‚Üí Pages
  2. Source: Deploy from a branch
  3. Branch: main / root
  4. Save
  5. Your URL: https://username.github.io/game-addressables/

Step 3: Upload Assets

  1. Build your Addressables (covered in next section)
  2. Copy contents of ServerData folder to repository root
  3. Commit and push: 
    git add .
git commit -m "Upload addressables build"
git push origin main
  4. Wait 2-3 minutes for GitHub Pages to deploy
‚ö†Ô∏è GitHub Limitations: Free tier has 1GB storage limit and 100GB bandwidth/month. Perfect for testing, but migrate to S3 for production.

Part 5: Configuring Remote Build Paths

Setting Up Profiles

Profiles let you switch between local testing, GitHub staging, and S3 production.

  1. Open Window > Asset Management > Addressables > Profiles
  2. You'll see "Default" profile. Create New Profiles :
    • GitHub Pages
    • AWS S3

Profile Configuration

GitHub Pages

RemoteBuildPath: ServerData/[BuildTarget]
RemoteLoadPath: https://yourusername.github.io/game-addressables/[BuildTarget]

AWS S3

RemoteBuildPath: ServerData/[BuildTarget]
RemoteLoadPath: https://your-bucket.s3.amazonaws.com/[BuildTarget]

üí° Pro Tip: Use the [BuildTarget] variable to automatically separate Android, WebGL, and StandaloneWindows64 builds into different folders.

Configuring Groups for Remote

  1. Select a group in Addressables window
  2. In Inspector, find "Build & Load Paths"
  3. Select "Remote"
  4. Enable "Build Remote Catalog" in Addressables Settings

Part 6: Building Addressables

Manual Build Process

  1. Select your target profile (Development/AWS S3/Github Pages)
  2. Go to Window > Asset Management > Addressables > Groups
  3. Click Build > New Build > Default Build Script
  4. Wait for build to complete
  5. Check ServerData/[BuildTarget] folder for output

Part 7: Uploading Your Addressables

Once Unity finishes building Addressables, you'll have a ServerData folder containing platform-specific bundles. Now it's time to upload these files to your chosen hosting platform.

üì§ Option 1: AWS S3 Upload

Your S3 bucket is already configured from Part 4. Now just upload the files:

üñ±Ô∏è Manual Upload (Beginners)

  1. Open your S3 bucket in AWS Console
  2. Click Upload
  3. Drag your ServerData/StandaloneWindows64 folder
  4. Repeat for Android and WebGL folders
  5. Keep folder structure intact
  6. Click Upload and wait for completion

URL Format: https://your-bucket.s3.amazonaws.com/[BuildTarget]

üì§ Option 2: GitHub Pages Upload

Perfect for development and testing. Your repository is already configured from Part 4.

üñ±Ô∏è Manual Upload (GitHub Web Interface)

  1. Navigate to your game-addressables repository
  2. Click Add file ‚Üí Upload files
  3. Drag contents of ServerData folder (maintain folder structure)
  4. Commit with message: "Update addressables - [date]"
  5. Wait 2-3 minutes for GitHub Pages to redeploy
  6. Access at: https://username.github.io/game-addressables/
‚ö†Ô∏è GitHub Limitations:
  • Repository size: 1GB maximum
  • Bandwidth: 100GB/month (soft limit)
  • File size: 100MB per file recommended
  • Best for: Testing and small-scale projects

‚úÖ Verification Checklist

  • All platform folders (Windows/Android/WebGL) uploaded
  • Folder structure matches Unity's build output
  • .json files have correct content type (application/json)
  • .bundle files have correct content type (application/octet-stream)
  • Files are publicly accessible (test URL in browser)
  • CORS headers configured correctly (for WebGL)
  • Unity's Remote Load Path matches upload URL exactly
‚úÖ Quick Test: Copy your Remote Load Path URL + /catalog_[hash].bin and paste in browser. If the JSON downloads, your setup is correct!

Part 8: Content Update Workflow (Hot Updates)

The most powerful feature of Addressables is updating content without republishing your app. Ship fixes, new levels, and balance changes instantly to all players.

üéØ Real-World Example: Fix a game-breaking bug in a prefab, rebuild only that bundle, upload it, and players get the fix within minutes‚Äîno app store approval needed.

Step-by-Step Update Process

1Ô∏è‚É£ Modify Your Assets

  • Change any Addressable asset (texture, prefab, audio, etc.)
  • Add new assets to existing groups
  • Update scriptable objects or config files

2Ô∏è‚É£ Check for Content Update Restrictions

  • Go to Window > Asset Management > Addressables > Check for Content Update Restrictions
  • Unity analyzes which assets changed since last build
  • A window shows all modified assets
  • Click "Apply Changes" to prepare them for update

3Ô∏è‚É£ Build Update

  • Go to Window > Asset Management > Addressables > Build > Update a Previous Build
  • Select your saved addressables_content_state.bin file
  • Unity builds ONLY the changed bundles
  • New bundles appear in ServerData folder
  • Catalog files (.json, .hash) are updated

4Ô∏è‚É£ Upload Changed Files

  • Upload the new/modified .bundle files
  • Replace the catalog_*.json and catalog_*.hash files
  • DO NOT delete old bundles‚Äîsome players may still reference them
  • Players on older app versions can still access old bundles

5Ô∏è‚É£ Players Automatically Update

  • Next time players launch, the game checks for catalog updates
  • Changed bundles download in background
  • No app store update required!

üìä Update Examples

üé® Art Update

Scenario: Fix a texture bug
Result: Only texture bundle (~5MB) downloads
Time: 2 minutes for players

üéÆ New Level

Scenario: Add new level content
Result: New level bundle (~20MB) downloads
Time: 5-10 minutes

‚öñÔ∏è Balance Patch

Scenario: Update ScriptableObject stats
Result: Tiny config bundle (~100KB) downloads

üîä Audio Hotfix

Scenario: Replace broken audio file
Result: Audio bundle (~3MB) downloads

üîÑ When Players Get Updates

Updates happen automatically when:

  • ‚úÖ Player launches the game
  • ‚úÖ Specific asset is loaded that has a newer version

Updates are downloaded:

  • üîπ In background (non-blocking)
  • üîπ Only for assets that changed
  • üîπ Can show progress bar (see Part 9)
  • üîπ Cached locally after download

‚ö° Rollback Strategy

If an update causes issues, you can quickly rollback:

  1. Keep previous catalog files backed up
  2. Re-upload the old catalog files to your server
  3. Players will download the previous version on next launch
  4. Fix the issue and push a new update

üéØ Hot Update Use Cases

  • üêõ Bug Fixes: Fix game-breaking issues immediately
  • üé® Art Updates: Improve visuals without resubmission
  • ‚öñÔ∏è Balance Changes: Adjust game difficulty

  • üó∫Ô∏è New Levels: Expand content post-launch

When to Use Addressables

Perfect For:

  • Games with large asset libraries (RPGs, open-world games)
  • Mobile games that need small initial downloads
  • Live service games requiring frequent content updates
  • Multi-platform releases (WebGL + Mobile + Desktop)
  • DLC and expansion content delivery
  • Games with localization into many languages

Consider Alternatives For:

  • Very small games (<50MB total) - overhead isn't worth it
  • Offline-only experiences - no benefit from remote loading
  • Rapid prototypes - setup time delays experimentation
  • Asset-light games (puzzle games, simple 2D platformers)
  • Projects with tight deadlines - learning curve exists
